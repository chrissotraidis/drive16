# Drive16 P1 Plan — quality loop state

Successor to `docs/2026-07-16-remediation-plan.md` (P0, merged in PR #2).
P0 made the system fast and safe; P1 makes the games good. Same loop
contract: work goals in order, one goal per iteration, run the acceptance
checks locally (Ollama qwen only for model calls), flip Status, commit with
the goal id.

Benchmark prompt (the user's bar): *a working Missile Command that looks and
sounds like a real Sega Genesis release — modern, good graphics, good music.*

Ground rules:

- Branch: `quality/2026-07-17-p1`. One commit per goal minimum.
- After any `app/src` change: `pnpm --dir app build`; contract + intent +
  watchdog checks must stay green.
- The benchmark runs through the real product path (browser app, AI sprites +
  MML music enabled) so what we audit is what a user gets.

---

## P1-G0 — Missile Command benchmark + output audit

Status: done — 2026-07-17. Benchmark run:
`artifacts/phase9/p1-benchmark/missile-command-1` (CLI, drive16-build agent,
sprites+music requirements, local qwen). 29.5 min, 20 steps (hit the cap),
741k input / 16.8k output tokens.

### Findings

1. **The game was never assembled — the root P1 defect.** The run produced
   two usable sprites (a readable alien ship and a genuinely good starburst
   explosion) and a **quality-passing soundtrack** (8 channels, 7
   instruments, 138 notes, percussion, 16.78 s loop — structurally the best
   this system has produced), then died at the step cap with **zero
   `build_rom` calls, zero emulator calls, `src/main.c` untouched**, all four
   file writes in scratch. Asset generation and gameplay assembly cannot
   share one 20-step budget. → P1-G1.
2. **`generate_sprite` exceeds the MCP client timeout on cold ComfyUI.** Two
   calls died as `-32001` while SDXL loaded (the generations actually
   finished server-side; the model re-generated, wasting ~5 steps). Split
   into fast enqueue/status/fetch calls and prewarm the checkpoint. → P1-G2.
3. **Music iteration happens in scratch and wiring never happens.** The
   model iterated `last.mml` via file edits + 4 compile calls but copying
   the VGM into `res/`, resource lines, `XGM_startPlay`, rebuild, and
   verify all still lay ahead when the budget died. The music pass must own
   compose→wire→verify end to end. → P1-G3.
4. **Discovery ate ~half the budget**: 13 reads + 6 globs + 3 RAG queries +
   a stray webfetch before any creation. Narrow per-pass instructions (and
   denying webfetch in build agents) fix this structurally. → P1-G1.
5. **Role/symbol mismatch**: the model generated an alien spaceship under
   the symbol `SPRITE_city_skyline`. Wiring assets by name is unreliable;
   the art pass should confirm role↔file mapping explicitly. → P1-G2.
6. Session-level defects found while launching the benchmark (already
   fixed): a single >300 s model response silently wedges the OpenCode
   session (fixed: 3,500-token step cap, contract updated); the in-app
   watchdog never fires in a throttled background browser tab (real fix in
   P1-G1: sequencing/watchdog move out of tab timers); overnight system
   sleep froze a run (long runs now wrapped in `caffeinate`).

Accept: all three boxes checked — ROM built+verified is carried by the
seeded scaffold (the agent's own build never happened — that IS the
finding), audit written, G1–G3 scopes refined below.

Run the benchmark through the app with AI sprites and MML music enabled.
Collect: gameplay frames (title + in-game via scripted Start), audio WAV,
ASSETS.md/PLAYTEST.md, sprite masters, run trace (steps/tokens/time).
Audit against the bar: Does it play? Do the graphics look 16-bit-finished
(sprites used? backgrounds? palette depth)? Does the music sound like a
Genesis track (parts, structure, length)? Where exactly does quality fall
short, and which pipeline limit causes each shortfall?

Accept:
- [ ] Benchmark ROM built and verified (screen/input/audio evidence).
- [ ] Output audit written into this plan under P1-G0 Findings, with each
      shortfall traced to a specific pipeline cause.
- [ ] P1-G1..G5 scopes refined from the findings before further work.

## P1-G1 — Phased build pipeline (implement → art → music → polish)

Status: done — 2026-07-17. Four pass-agents added to `opencode.json`
(implement 20 steps/temp 0, art 16/0.4, music 16/0.4, polish 12/0, all
capped at 3,500 output tokens with per-pass tool permissions) and
`scripts/run-phased-build.mjs`: sequential bounded sessions over one project,
deterministic gates (fresh ROM + screen liveness + audio; the full
presentation contract is recorded as information because it false-positives
on dense patterned backgrounds), one bounded continuation when a pass dies
at its step cap mid-fix, evidence stamping between passes, per-pass ledger.

Acceptance run (`artifacts/phase9/p1-benchmark/missile-command-phased`,
attempt 3 after two diagnostic attempts that each produced a runner fix):
**all gates green in 44 min** — implement 261 s; art wired 3 role-correct
generated sprites (attacker/explosion/cities, 19 `SPR_*` calls) with one
continuation; music composed and wired `missile_theme.vgm` first try with
non-silent audio verified; polish finished with one continuation. The
in-game frame shows a real Missile Command: HUD (score/wave/cities),
crosshair, generated city sprites, textured ground, control legend — the
exact assembly the P1-G0 single-pass run never reached. Learnings folded
into the runner: sprite-API crib (saves ~6 steps of rediscovery),
build-by-step-12 compactness rule, liveness-vs-presentation gate split.

Benchmark-proven: one 20-step budget cannot hold gameplay + art + music
(P1-G0 finding 1). Split the build into sequential bounded passes over the
same project folder, sequenced by a deterministic runner (not tab timers —
finding 6):

- New opencode.json agents: `drive16-implement` (temp 0, gameplay to first
  verified build), `drive16-art` (temp 0.4, generate/wire/verify sprites,
  webfetch/rag denied), `drive16-music` (temp 0.4, compose/wire/verify
  soundtrack end to end), `drive16-polish` (temp 0, fix what verification
  flagged). Each with its own steps/token caps; every pass ends with
  build_rom + evidence or an explicit blocker.
- `scripts/run-phased-build.mjs`: seeds/uses a project, runs the passes as
  separate `opencode run` sessions with narrow per-pass prompts, stops on a
  failed gate, stamps mechanical evidence between passes, records a
  per-pass ledger (time/tokens/result). Used by benchmarks AND callable as
  the app's build backend later; app UI integration may land separately.

Accept:
- [ ] Agents + runner exist; contract updated deliberately where it pins the
      single-pass shape.
- [ ] A phased run on a fresh Pong or Missile Command project with
      sprites+music produces: agent-built ROM (fresh), wired generated
      sprite(s), wired quality-passing soundtrack, verified screen/input/
      audio — the exact assembly P1-G0 never reached.
- [ ] Per-pass ledger recorded; no pass exceeds its budget.

## P1-G2 — Asset pipeline v2 (crops, candidates, backgrounds)

Status: done — 2026-07-17. The comfyui MCP server gained a single-flight job
model: `enqueue_sprite` / `enqueue_background` return a job id instantly,
`job_status` / `fetch_result` never block — verified against a genuinely
cold ComfyUI (models force-unloaded via /free): every call stayed under 10 s
while the cold generation ran minutes in the background, and the finished
sprite was fetched with its role intact. Fixed a zombie-reaping bug found by
the test (child exits were invisible to `os.kill(pid,0)`; now tracked
Popen + defensive `waitpid`). `role` is recorded end-to-end
(tool arg → job → result → ASSETS row guidance in the art pass).
New background path: `run-comfyui-background-workflow.py` +
`validate-generated-background.py` produce plane-sized 16-color indexed PNGs
(verified 256×128/P/16); a generated backdrop wired as `IMAGE ... BEST` in a
pong-basic copy **builds clean**. `prewarm-comfyui-checkpoint.py` (1-step
64×64 render) hooked into `prewarm-local-tools.sh`. Deferred to P2 with
cause: crops-from-master derivation and batch candidate ranking — the
benchmark showed timeout, role mapping, and the missing background path were
the binding constraints; candidates/crops are enhancements on top of a now
timeout-proof base.

Reprioritized by P1-G0 findings 2 and 5:

1. **Timeout-proof generation**: split `generate_sprite` into fast calls —
   `enqueue_sprite` (returns job id immediately), `sprite_status(job)`,
   `fetch_sprite(job)` — so no MCP call outlives a cold SDXL load; keep the
   synchronous tool for warm paths with a conservative internal budget.
   Prewarm the checkpoint (readiness fires a tiny 4-step render).
2. **Explicit role↔asset mapping**: tool takes a `role` field recorded in
   the result and stamped into ASSETS.md rows so a spaceship can't get wired
   as a city skyline.
3. Then the original scope: derive crops from the preserved 512px master;
   batch candidates + auto-rank; tileset/background path through rescomp
   IMAGE/TILESET.

Accept:
- [ ] Cold-ComfyUI generation completes without `-32001` (verified by
      unloading the model first).
- [ ] Role recorded end-to-end; a generated background/tileset lands in a
      test project and builds.

## P1-G3 — Music pipeline v2 (templates, dedicated budget, listen loop)

Status: done — 2026-07-17. Four verified-good genre templates in
`assets/enhancements/mml/genre-templates/` (action-tension from the
benchmark composition, plus authored bouncy-arcade, puzzle-groove,
adventure-lite): every one compiles clean with `quality.pass` true
(7 channels incl. PSG-noise percussion, 50–65 s loops), repeatably checked
by `scripts/verify-mml-templates.py`. The pong/snake/tetris skeleton stubs
(identical 159-byte blips) are replaced with real 11–13 KB tracks compiled
from the matching templates, and the presentation baseline (incl. audio
checks) still passes. The music pass prompt now starts from the closest
template as few-shot with a 4-attempt compile budget. Debugging on the way
in exposed two structural-gate quirks worth knowing: the analyzer only reads
lines that start with a channel letter (fixed by normalizing templates to
letter-per-line style, which the pass prompt now teaches), and the P1-G1
music pass's "passing" `last.mml` was actually its final failed attempt —
the wired VGM came from an earlier attempt (stray bare `t140`, removed).

Three of five skeletons ship a 159-byte stub; the gate counts notes, nobody
listens — and P1-G0 finding 3 showed composition succeeding in scratch while
wiring never happened. The music pass (P1-G1) owns compose→wire→verify end
to end. This goal adds the material: 3–5 verified-good genre MML templates
(corpus + few-shot in the music-pass prompt — the benchmark's quality-passing
"Skyline Intercept Theme" is candidate #1); replace the stub VGMs in
pong/snake/tetris skeletons with real short tracks compiled from those
templates; music pass compile budget ≥4 attempts plus a
render→feature-check→revise-once loop.

Accept:
- [ ] Template library compiles clean with `quality.pass` true for each.
- [ ] Skeleton stub VGMs replaced; presentation baseline still passes.
- [ ] Music pass in P1-G1 uses templates as few-shot and survives 2 syntax
      errors without abandoning music.

## P1-G4 — Fast iteration contract

Status: done — 2026-07-17. New `drive16-iterate` agent (12 steps, temp 0,
sprite/music/rag/webfetch denied) is now the route for feature follow-ups on
a working game (classifier updated + unit tests; repair stays for
broken-game prompts). Iteration-mode prompt scopes verification to the
change (audio checks only when the request touches sound; docs untouched —
the trace generator stamps evidence). Live in-app acceptance: "make the ball
move a little faster and the paddles one tile taller" on the active Pong →
**1 m 49 s** send-to-finish on local qwen with the exactly-right diff
(`BALL_FRAMES 8→6`, `PADDLE_H 4→5`), fresh ROM, screen-alive verified
(9 colors). One wrinkle found and fixed: the agent's bare `build_rom` tool
names resolved as invalid (the app's deterministic rebuild safety net
covered it); the iteration prompt now names the prefixed MCP tools.

Iteration measured at ~3 min via harness; the in-app path re-runs the full
verification gauntlet. Scope verification to the change (code-only edit →
build + screen diff + input probe; asset change → add audio/asset checks),
reuse the running emulator state where possible. Target: "make the paddles
bigger" lands in ≤2 min in-app.

Accept:
- [ ] In-app follow-up on an existing game completes in ≤2 min wall-clock
      with correct diff and scoped verification evidence.
