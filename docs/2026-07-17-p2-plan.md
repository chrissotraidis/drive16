## P2-G0 — Reference feel profiles from real Genesis games

Status: done — 2026-07-17. `scripts/profile-reference-rom.py` plays ROMs
through Genteel with scripted controllers (genre-shaped input scripts in the
TAS format), streams frames + audio, and measures the feel profile. All
three user-supplied references profiled — sample frames prove real gameplay
was reached (Green Hill Zone, Round 1 street fight) — plus the P1 build with
the same instrument. Result (`docs/genesis-feel-bar.md`): the gap is two
orders of magnitude on motion/liveliness (Sonic: 40–233 moving regions, 30%
pixel churn, 44% camera-motion share; our build: 0–1 regions, 0.06% churn,
87% static screen). Numeric bar adopted for P2-G1..G3. Honest gap: audio
onset measurement under-discriminates against music beds and the Genteel WAV
dump shows zero high-band energy for every game — capture/measure chain
investigation moved into P2-G2 before audio targets are set.

Run real Genesis ROMs headless through Genteel (extending
`capture-reference-run.py`) and extract a measurable feel profile per game:
pixel-velocity histograms, moving-object counts, audio transient density
(SFX events/sec), animation cadence, palette counts, HUD share, scroll-plane
deltas. Sources: ROM(s) the user supplies from their own collection dropped
into `artifacts/reference-roms/` (behavior analysis only, nothing extracted
or committed), plus permissively-licensed homebrew as a fallback. Output:
`docs/genesis-feel-bar.md` with the numeric bar (e.g. "median moving object
travels 1–3 px/frame at 60 Hz; ≥1 audio transient within 100 ms of every
player action; explosions animate ≥4 frames").

Accept:
- [ ] Profiles captured for ≥2 real games; the P1 Missile Command profiled
      with the same tooling shows the gap numerically on every dimension.
- [ ] `docs/genesis-feel-bar.md` written with hard targets adopted by later
      goals.

## P2-G1 — Game-feel library + skeletons rebuilt on it

Status: todo — NEXT UP. Strategic reframe (2026-07-17, after P2-G0): the
model reproduces whatever floor it is given; it composes well but cannot
invent engine-feel inside a bounded budget. So the fix is deterministic C
engineering: a small shared feel library in the starter, with the model's
job changed from "write an engine" to "compose a game from parts."

**P2-G1a (first loop, ~1–2 h):** create `examples/app-starter-blank/src/`
feel modules (shipped with every project): fixed-point velocity/position
movement with easing (`move_toward`, accel/decel), an SFX bank with one-call
triggers (`sfx_play(SFX_SHOT)` — PSG envelope effects, no PCM needed at
first), a sprite animation player (`anim_play`, 4-frame explosion), screen
shake + hit-pause helpers, and a custom HUD tile font replacing
`VDP_drawText` for game-facing text. Rebuild the missile-command skeleton on
the library. Exit: `profile-reference-rom.py` scores it at the bar (moving
regions ≥8 median, churn p90 ≥0.05, static rows ≤0.20, per-pixel motion) and
the build is loaded into the app player for the user's taste review — the
user's verdict gates the loop, not the profiler.

**P2-G1b (second loop):** spread the library to pong/asteroids skeletons,
add plane-B parallax to two skeletons, and teach the implement/art/polish
pass prompts the library vocabulary so generated games call it.

Accept:
- [ ] G1a: rebuilt missile-command meets the numeric bar; user review
      collected on a playable in-app build.
- [ ] G1b: two more skeletons on the library with parallax; pass prompts
      updated; presentation baseline + all existing checks stay green.
