## P2-G0 — Phased builds in the app UI

Status: todo

The app still runs the old single bounded pass; P1 quality only exists via
`scripts/run-phased-build.mjs`. Make the phased flow the app's build path for
new games when enhancements are enabled:

- `runAgentPrompt` sequences implement → art → music → polish as separate
  OpenCode sessions over the active project (reusing the pass prompts and
  per-pass agents from the runner — extract them to a shared module so the
  CLI runner and the app import one source of truth).
- Deterministic gates between passes run through the existing browser
  verification endpoints; a failed gate triggers the pass's single bounded
  continuation, then stops honestly with the blocker in chat.
- Build log groups events by pass with a status chip per pass
  (implement/art/music/polish: running/ok/failed); chat gets one short
  message per pass transition, not a wall.
- Single-pass path remains for iterations (drive16-iterate) and for games
  with enhancements off.

Accept:
- [ ] In-app "make a missile command game …" with sprites+music enabled runs
      all four passes (trace evidence per pass), ends with wired generated
      sprites + verified soundtrack, and the game visible in the player.
- [ ] A failed gate surfaces its blocker in chat instead of a silent stall.
- [ ] tsc + contract + intent + watchdog green; pass prompts shared between
      app and CLI runner (no fork).

## P2-G1 — Self-play playability scoring

Status: todo

Screen liveness proves pixels; nothing yet proves *play*. Build a generic
self-play probe: scripted input runs (directional sweeps, action presses,
restart) measuring HUD-region change (score movement), game-over/gameplay
state transitions, and frame-diff entropy across the run. Output
`playability-score.json` per project; wire it into the polish gate as
information and into the promote/review flow as evidence.

Accept:
- [ ] Probe scores the four skeleton games and the P1 Missile Command build
      with sane, discriminating numbers (e.g. attract screen scores lower
      than active play).
- [ ] Polish pass ledger includes the score; a zero-interaction game is
      flagged.

## P2-G3 — Asset & art polish

Status: todo

Carried from P1-G2 with additions:

- Batch sprite candidates (N=3 per role) ranked automatically (validator
  metrics + coverage/sharpness heuristics); winner wired, all three kept on
  disk for the user.
- Derive crops from the preserved 512px master (16/24/48px variants) so one
  diffusion yields a size family.
- Two-frame animation support: generate frame pairs, build a 2-frame SPRITE
  resource, `SPR_setAnim` in the art-pass crib.
- Art pass wires a generated plane-B background by default when the scene
  suits it (the P1-G2 background path exists but is optional today).

Accept:
- [ ] One diffusion call yields ranked candidates + crop family on disk.
- [ ] A game ships with an animated 2-frame sprite and a generated
      background, builds clean, passes presentation baseline.
